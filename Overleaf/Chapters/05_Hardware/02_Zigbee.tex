\section{Wireless Communication and Network}
 
In a wind turbine installation comprising several thousand bolts and joints \cite{6000bolts}, each fitted with its own module, the network must support a large number of devices while remaining robust under the challenging RF conditions imposed by the metallic structure of the turbine.

A mesh topology addresses both requirements directly. Rather than relying on a single central node that every module must reach, a mesh allows modules to relay messages for one another, so data travels over several short hops and coverage extends across the structure even where the metal blocks a direct radio path. The same multiplicity of paths makes the network self-healing, since traffic can be rerouted around an obstructed link or an unresponsive node. The established low-power mesh protocols and the microcontrollers that support them operate predominantly in the globally available \SI{2.4}{\giga\hertz} band, which makes it the natural choice for a standards-based mesh deployment. Lower frequencies propagate more effectively through obstructions such as the turbine's metal structure, but sub-\si{\giga\hertz} operation falls outside this established mesh ecosystem, so the candidate protocols are drawn from the \SI{2.4}{\giga\hertz} band.

Among comparable \SI{2.4}{\giga\hertz} mesh protocols, the primary candidates are Zigbee, Thread, and Bluetooth Mesh, the latter built on Bluetooth Low Energy (BLE). Bluetooth Mesh extends BLE, which on its own targets short-range peer-to-peer links, to multi-hop networking, but it is a comparatively young technology, standardized in 2017 and still maturing. Zigbee, by contrast, has been deployed since 2004 across hundreds of millions of nodes and remains the most widely used mesh protocol \cite{Avnet_zigbee_ble}. This maturity is a meaningful advantage for a system intended to operate unattended for more than ten years, where a proven and well-documented stack reduces the risk of latent firmware issues over the service life. Zigbee also scales further, with a theoretical ceiling near 65,000 nodes against roughly 32,000 for Bluetooth Mesh. The principal point in favour of Bluetooth Mesh is its lower active power draw, and for a strictly power-constrained design it would be worth investigating further as a future alternative. The node spends nearly all of its time asleep, however, so this difference has little effect on the overall energy budget.

Thread provides native IP-based connectivity, which simplifies integration with internet and cloud services but requires a larger and more complex software stack, increasing flash and RAM usage \cite{Thread_vs_zigbee}. Both Zigbee and Thread can run on the same class of \SI{2.4}{\giga\hertz} wireless microcontroller, but for a node that wakes only briefly to transmit a single measurement before returning to sleep, this additional capability offers limited benefit relative to its cost in firmware footprint. Zigbee instead employs a lighter, purpose-built mesh layer with lower packet overhead and simpler handling of sleepy end devices, permitting more compact and cost-effective hardware \cite{Zigbee_alliance}.

On the strength of its maturity, compact firmware footprint, and efficient low-power mesh operation, Zigbee is selected as the wireless protocol for this work. The resulting network, illustrated in \cref{fig:Zigbee_Network}, is formed from three node types. End devices represent the sensor modules, routers relay messages between nodes, and a coordinator manages the network and forwards the collected data to the server application

\subsection{Integrated versus External Zigbee Module}
 
The wireless communication circuit can either be integrated directly onto the module PCB or implemented using a pre-certified external RF module. The primary advantage of an external module is reduced design complexity, as the RF circuitry and antenna matching are already optimized and validated by the manufacturer, lowering the risk of RF performance issues and broadening the choice of microcontroller. However, external modules increase both unit cost and PCB footprint, and may introduce additional integration considerations. The integrated approach was therefore selected, since the compact footprint demanded by the module geometry and the unit cost, multiplied across the thousands of modules in a single turbine, outweigh the reduced design effort an external module would provide.

\subsection{The Zigbee Network}
\label{sec:zigbee_network}

The Zigbee network implemented in a wind turbine is illustrated in \cref{fig:WC_Overview}. The end devices, represented by yellow dots, are placed on each bolt of a flanged joint and transmit measurement data to the nearest router. The routers, represented by blue dots, are distributed throughout the turbine tower to provide coverage and relay data toward the coordinator. The coordinator, represented by the green dot, collects data from all end devices and forwards it to a server application via a WiFi connection.
 
Each end device is equipped with a \SI{2.4}{\giga\hertz} antenna for communication with the routers. The routers must be mains-powered devices, as they remain active continuously to relay traffic. In practice they can be any permanently powered device with Zigbee support, such as smart outlets or similar infrastructure. In the prototype setup, a Sonoff Zigbee 3.0 USB Dongle Plus is used as the coordinator \cite{sonoff_zbdongle_p}, connected to a computer running the open-source Linux distribution Ubuntu, which hosts the user-facing software described in \Cref{sec:Software_imp}. The router infrastructure is not explored further in the prototype.
