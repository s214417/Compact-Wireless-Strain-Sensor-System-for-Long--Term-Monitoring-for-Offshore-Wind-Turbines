\section{Software Implementation (Graphical User Interface)} \label{sec:Software_imp}

The user-facing layer of the system is implemented in Home Assistant, an open-source automation platform that runs locally on a server and requires no cloud connectivity. Home Assistant is chosen for the proof of concept because it provides three capabilities that would otherwise have to be developed from scratch. Home Assistant has a native Zigbee Home Automation (ZHA) integration that exposes the sensor node without any custom middleware, a built-in recorder that logs every measurement to a local time-series database, and a configurable dashboard framework for presenting the data. This allows the work to focus on the monitoring concept rather than on backend infrastructure, while keeping all data on-site, as is appropriate for an offshore installation served onshore.

When a node joins the network as described in \cref{sec:Zigbee_Network_Management}, ZHA enumerates it automatically and exposes each of its reported quantities as an individual Home Assistant \emph{entity}. The recorder stores the history of these entities and the dashboard reads their current state.

\subsection{Dashboard Structure}
The dashboard is a Lovelace dashboard defined in YAML. Its layout deliberately mirrors the physical hierarchy of the installation, so that the on-screen structure matches the way an operator reasons about the plant. Front page is a park overview containing several turbines, each turbine has a number of bolted flange joints, and each flange is secured by a ring of nuts, with one sensor node per monitored bolt. This four-level hierarchy is illustrated in \cref{fig:gui_hierarchy}.

Navigation follows the same hierarchy as a drill-down. Tapping a turbine opens a sub-view showing its flanges, and tapping a flange opens a sub-view showing its individual nuts. The operator therefore starts from a single park-level overview and descends only into the branch that requires attention, rather than being presented with every sensor at once.

\subsection{Status Visualisation}
Each card is colour-coded as a traffic light to communicate condition at a glance. A card compares its value against two thresholds, a warning level and a critical level, and is rendered green, yellow, or red accordingly.

The essential feature of the dashboard is that every parent card shows a worst-case roll-up of the sensors beneath it. A turbine card takes the most severe reading among all nuts on all of its flanges, and a flange card the most severe among its own nuts. A single nut entering the critical range therefore turns its flange and its turbine red as well, so the condition is visible already on the park overview, and the colour trail guides the operator down to the affected bolt. This keeps the park view informative even for a large number of nodes, where individual readings could not all be shown at once.

The dashboard is built from two community custom cards: \texttt{button-card}, which renders the individual cards and evaluates the threshold and roll-up logic, and \texttt{layout-card}, which arranges them in responsive grids that reflow for narrower screens. Shared styling and logic are factored into a small set of \texttt{button-card} templates (\texttt{shm\_base}, \texttt{shm\_turbine}, \texttt{shm\_flange}, and \texttt{shm\_nut}), and YAML anchors are used to avoid repeating the entity lists for each turbine and flange. The complete dashboard configuration is provided in Appendix~B.

Besides the strain reading, each nut card surfaces basic node-health information to support maintenance of a wireless, battery-powered network. Underneath the strain reading is the on-board temperature, which the node already includes in its report. Besides, the radio signal strength (RSSI) and the time since the node last reported, updated or changed, is both obtained directly from Home Assistant, and displayed in the nut card.
Battery state is not currently available, as the on-node battery monitoring would require a change to the PCB and is therefore treated as a future improvement in \cref{sec:Discussion}.

As the prototype comprises a small number of physical nodes, its entities are reused across the modelled park to populate the turbines, flanges, and nuts, so that the interface can be demonstrated at a realistic scale. Screenshots of the GUI dashboard is seen in \cref{fig:GUI}.
