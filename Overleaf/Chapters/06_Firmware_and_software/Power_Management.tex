The EFR32MG22E provides six energy modes, EM0, EM1, EM1P, EM2, EM3, and EM4, ordered from the full-performance run mode EM0 down to the deepest sleep mode EM4, where each successive mode disables more of the device to save power at the cost of reduced functionality and a longer wake-up.
Normally a power optimised design moves through several of these modes within a cycle, dropping to an intermediate sleep mode whenever the processor is briefly idle.
This firmware does not do so, and instead uses only two of the six modes, namely EM0 during the active phase and EM4 during the long sleep between cycles.

EM4 is used for the sleep phase because it consumes the least power of any mode and the device spends almost all of its time there, where a single sleep interval could last as long as a month.
Since the deep-sleep current sets the battery lifetime far more than the brief active current does, using the deepest available mode for this phase is the single most important power decision in the firmware.
EM0 is used for the entire active phase because that phase is deliberately kept as short as possible and is dominated by Zigbee communication with Home Assistant, which requires the radio and the processor to be fully active and therefore cannot proceed in a lower mode.
Introducing intermediate modes such as EM2 or EM3 into the active phase would complicate the run cycle while saving very little, since the idle gaps within an active phase that lasts only a few seconds and is mostly spent communicating are too short and too few to yield a meaningful energy reduction.
The design therefore optimises the active phase by making it brief rather than by switching modes within it.

The wake-up from EM4 is generated by the Backup Real-Time Counter (BURTC), which is one of the few peripherals that remains operational in EM4 Shutoff.
On wake-up the device undergoes a full reset rather than resuming execution, so the firmware does not need to preserve any state across sleep, as discussed in \cref{sec:Measure-Report-Sleep_Cycle}.
In this firmware the BURTC is driven in EM4 from the ULFRCO, the internal ultra-low-frequency RC oscillator running at approximately 1\,kHz, which is deliberately kept running as the EM4 group-A clock source while the more accurate low-frequency oscillators are disabled to minimise EM4 current.
The ULFRCO is a low-power internal oscillator with a tolerance of a few percent, so the wake timing carries a corresponding few-percent jitter, which is entirely acceptable for a reporting cycle measured in minutes to a month.
To enter sleep, the firmware applies this oscillator configuration and parks the bridge GPIO pins as described in \cref{sec:Sensor_Acquisition} so that no leakage current flows during sleep.

The duration of the sleep is set by programming a BURTC compare value at which the counter triggers the wake.
A consideration specific to EM4 is that the rate at which the BURTC counter runs itself affects the current drawn in EM4, where a slower counter draws less.
The firmware exploits this by selecting the BURTC prescaler according to the requested sleep length, using a fine counter tick for short sleeps where accurate timing matters and a coarse tick for long sleeps where the lowest possible current is the priority.
The prescaler selection is shown in \cref{lst:burtc_clkdiv}, where the divider is chosen in coarse steps from a one-millisecond tick for sleeps up to a minute through to a tick of tens of seconds for sleeps of a day or longer.

\begin{lstlisting}[language=C, caption={BURTC prescaler selection in \texttt{em4.c}, trading wake-timing granularity for lower EM4 current on longer sleeps.}, label={lst:burtc_clkdiv}]
static uint32_t choose_burtc_clkdiv(uint32_t sleep_ms)
{
    if (sleep_ms <= (60U * 1000U)) {            // <= 1 min
        return 1U;      // ~1 ms tick @ ~1 kHz ULFRCO
    }
    if (sleep_ms <= (10U * 60U * 1000U)) {      // <= 10 min
        return 1024U;   // ~1 s tick
    }
    if (sleep_ms <= (24U * 60U * 60U * 1000U)) { // <= 24 h
        return 8192U;   // ~8 s tick
    }
    return BURTC_EM4_CLKDIV_LONG;               // ~33 s tick
}
\end{lstlisting}

Once the prescaler is chosen, the number of counter ticks corresponding to the requested sleep is computed from the actual EM4 group-A clock frequency and the selected divider, so that the wake timing stays correct regardless of which prescaler bucket applies.
The tick count follows from the requested duration in milliseconds, the clock frequency, and the divider, and is rounded up so that the device never wakes early, as shown in \cref{eq:burtc_ticks} where $f_{\mathrm{clk}}$ is the EM4 group-A clock frequency and $d$ is the BURTC divider.

\begin{equation}
\label{eq:burtc_ticks}
N_{\mathrm{ticks}} = \left\lceil \frac{t_{\mathrm{sleep}}\,[\mathrm{ms}] \cdot f_{\mathrm{clk}}\,[\mathrm{Hz}]}{1000 \cdot d} \right\rceil
\end{equation}

This computed value is loaded into the BURTC compare register before the device enters EM4, after which the BURTC counts up from reset and the compare match wakes the device through the EM4 wake-up path once the requested interval has elapsed.
